# Session-Scoped Terminal Integration Design

**Date:** 2026-05-28
**Status:** Draft, awaiting user review
**Mockup:** `agentry.pen` → frames `Terminal Option A — Bottom Panel` (id `fVGgS`), `Terminal Option B — Right Sidebar Tab` (id `ihgAN`), `Terminal Option C — Top Tab Switch` (id `qRiqR`) — **C 是最接近最终方案的**（"终端替换 chat 工作区"），但实际去掉了 tab strip，改成 topbar 右侧单一 toggle 按钮（见下面 UI Behavior 一节）。

## Goal

在 session 详情页加入"会话感知"的终端能力：每个 session 顶部多一个终端 toggle 按钮，点开 → spawn 一个 PTY 替换 chat 视图、cwd 自动落在 session 的工作目录；点关 → 立即 kill PTY。本地 session 走本地 `creack/pty`，远端 session（`Backend.DeviceID` 非空）走 agentred 上的 PTY，前端对底层透明。

终端完全 ephemeral——按钮 OFF 期间无任何后台 process、无 scrollback、无 audit log。这一版的差异化定位是"session 感知"（自动 cwd、自动选本地/远端机器），而不是要做一个通用 terminal multiplexer。

**非目标（MVP 明确不做）**：
- Windows 支持（PTY 走 ConPTY，差异面大；先 macOS/Linux）
- 多终端（一个 session 同时只能有一个活 PTY）
- 持久化（scrollback、命令历史、audit log 都不入 SQLite）
- Local echo（远端 latency 在 LAN 场景可忽略，复杂度不值）
- 终端 tab 复用（再次 Open = 全新 shell，不恢复上次）

## Architecture

```
Frontend (xterm.js in session-detail page)
    │  Wails service calls + EventsOn subscriptions
    ▼
internal/service/terminal_svc/        ← 新增 Wails-bound service
    │  - Open / Write / Resize / Close
    │  - sessionID → Handle map（mutex 守护）
    │  - 根据 session.Backend.DeviceID 选 backend
    ├──────────────────┐
    │                  │
    ▼                  ▼
internal/pkg/pty/local      internal/pkg/pty/remote
(creack/pty 包一层)         (agentred ws client 包一层)
                                  │
                                  ▼
                            agentred / internal/daemon/handlers/terminal.go
                                  │
                                  ▼
                            creack/pty (on remote box)
```

**关键约束**

- 每个 PTY Handle 用 `sessionID` 做唯一 key——同一 sessionID 重复 Open 会先 Close 旧 Handle 再建新的（语义：用户重新点开按钮 = 全新 shell）。
- Wails event 通道名 `terminal:<sessionID>:data` 和 `terminal:<sessionID>:exit`——namespacing 到 sessionID，多 session 切换不会串。
- agentred 的 `terminal.data` push 复用现有 emitter pattern（`internal/daemon/handlers/session.go` 现有 streaming 写法）。
- PTY 进程：
  - `cwd` = `chat_svc.resolveSessionCwd(session)` 现有函数返回值
  - `env` = 当前 agentre / agentred 进程的 env + `TERM=xterm-256color`
  - `shell` = `$SHELL` 否则 `/bin/sh`

**前端不知道远端**：单一 Wails service + 单一 event channel pattern，方案 1（backend dispatch）的核心。

**两个 interface 的职责切分**

- `pty.Handle`（`internal/pkg/pty/pty.go`）—— 一个已存在的 PTY 实例的运行时 API：`Write(data)` / `Resize(cols, rows)` / `Close()` / `Exit() <-chan ExitInfo` / `Data() <-chan []byte`。本地和远端 backend 都实现这个；service 层只跟 Handle 打交道，不关心进程在哪台机。
- `terminal_svc.PTYBackend`（`internal/service/terminal_svc/backend.go`）—— factory，只一个方法：`Open(ctx, cwd, env, shell, cols, rows) (pty.Handle, error)`。`local.Backend` 和 `remote.Backend` 是它的两个 impl。service 在 `Open()` 时根据 session 选 backend，拿到 Handle 之后剩下都走 Handle。
- 隔离动机：测试时 mock `PTYBackend` 模拟 spawn 失败 / 模拟 Handle 行为；real impl 之间没有"是不是用 ws"这种状态泄露到 service。

## Data Flow

**用户点开终端按钮**

1. 前端：`Terminal.Open(sessionID, cols, rows)` → service 查 session → 选 backend → spawn PTY → 返回 `{ok}`
2. 前端：`EventsOn("terminal:<sid>:data", cb)`；xterm 把 cb 拿到的 data `write()` 进去
3. 用户敲键 → xterm `onData(str)` → `Terminal.Write(sessionID, str)` → backend PTY stdin
4. 窗口/字号变 → xterm fit addon → `Terminal.Resize(sessionID, cols, rows)` → PTY.Setsize
5. 用户再点按钮关闭 → `Terminal.Close(sessionID)` → SIGHUP → 200ms grace → SIGKILL 兜底 → emit `terminal:<sid>:exit`
6. PTY 自然退出（user 敲 `exit`）→ Go reaper goroutine 收到 EOF → 同样 emit `exit` + 清 map

## Protocol / Wire Format

### Wails service（前端 ↔ Go 桌面端）

```go
package terminal_svc

func (s *Service) Open(sessionID int64, cols, rows uint16) error
func (s *Service) Write(sessionID int64, data string) error
func (s *Service) Resize(sessionID int64, cols, rows uint16) error
func (s *Service) Close(sessionID int64) error
```

- `Open` 重入：已有 Handle 先 Close 再建新的（不复用）
- `Write.data` 是 raw（xterm `onData` 给的就是 string，含 ANSI/控制字符），Go 端 `[]byte(data)` 直接喂 stdin
- 失败语义：`ErrSessionNotFound` / `ErrDeviceOffline` / `ErrTerminalClosed` / spawn 错误透传

### Wails events（Go → 前端 push）

| event 名 | payload | 时机 |
|----------|---------|------|
| `terminal:<sessionID>:data` | `{ "data": "<raw string>" }` | PTY stdout chunk |
| `terminal:<sessionID>:exit` | `{ "code": int, "reason": "natural\|killed\|connection_lost\|daemon_shutdown\|error", "msg": "..." }` | PTY 退出 |

- 不做 base64（Wails JSON 编码处理 UTF-8 escape 即可，xterm 处理 byte 序列鲁棒）
- chunk 大小上限 8KB，按 PTY 输出节奏 push（不按行——避免 vim/htop 类应用乱）

### agentred JSON-RPC（桌面 ↔ 远端 daemon）

参考 `internal/daemon/handlers/session.go` 现有 handler 形态。

**Methods（client → daemon）**

```go
type TerminalOpenParams  struct{ SessionID int64; Cwd, Shell string; Env []string; Cols, Rows uint16 }
type TerminalOpenResult  struct{ TerminalID string }  // daemon 生成的 uuid，桌面端只透传

type TerminalWriteParams  struct{ TerminalID string; Data string }
type TerminalResizeParams struct{ TerminalID string; Cols, Rows uint16 }
type TerminalCloseParams  struct{ TerminalID string }
```

**Push events（daemon → client）** — 复用现有 ws push 通道：

```go
type TerminalDataEvent struct{ TerminalID string; Data string }
type TerminalExitEvent struct{ TerminalID string; Code int; Reason, Msg string }
```

桌面端 `remote.Handle` 收 push 后转成本地 Wails event（按 sessionID 命名，对前端透明）。

### 背压

daemon 侧 PTY → ws 之间 channel buffer 256 chunks。满时 **丢旧 chunk + 在丢弃位置插一条 `[--- output throttled ---]` 标记**，不阻塞 PTY stdout（避免远端进程因为我们 ws 慢而 stdout block，造成死锁）。

### 字符编码

PTY 输出可能含任意 byte（中文 UTF-8、ANSI escape、二进制 escape 序列）。Wails events JSON 编码不会破坏合法 UTF-8；非法 byte 序列罕见，走 Latin-1 fallback。

## UI Behavior

### 按钮位置

挂在 chat 区域 topbar 右侧（`frontend/src/components/agentre/chat-page.tsx` 当前 topbar 结构）：

```
[ Session Title    … other badges … ]  [▢ 终端]  [⋯ More]
                                         ↑ 新增
```

### Toggle 状态机

```
              click (Open 异步开始)
   IDLE  ─────────────────────────►  OPENING
    ▲                                   │
    │                                   │ Open() resolves
    │ Close() resolves                  │
    │  + exit event                     ▼
   CLOSING  ◄──────────────────  OPEN  (PTY alive, xterm shown)
                                   │ ▲
                  click /          │ │  PTY 自然 exit
                  Close() invoke   │ │  → exit event → 自动回 IDLE
                                   ▼ │  （exit 0 安静、非 0 toast）
```

| 状态 | UI | 按钮 | 切走 session/页面 |
|------|-----|------|----|
| IDLE | chat 视图 | 灰色 outline | n/a |
| OPENING | chat 视图 + 顶部 spinner | disabled | store 标记 abort，Open resolve 后立刻 Close |
| OPEN | 终端视图（替换 chat） | filled 高亮 | 立即 `Close()` + unmount + EventsOff |
| CLOSING | 终端视图 + 半透明遮罩 | disabled | n/a |

### 切换语义

- **再点按钮 ≠ 隐藏**：真正 Close，PTY 立即 kill；再开就是 fresh shell。
- **session/tab 切换**：切到别的 tab 不影响已开的 PTY —— PTY 跟随 ChatPanel 的 mount 生命周期。`ChatPanelHost` 用 `display:none` 隐藏非活动 tab，ChatPanel 仍 mounted，TerminalPanel 也 mounted，PTY 继续跑。用户在 A 跑 `npm test` 切到 B 看代码再切回 A，输出还在。
- **tab 关闭 = kill PTY**：关闭 tab 触发 ChatPanel unmount → TerminalPanel unmount → `useTerminal` cleanup 调 `Terminal.Close` → 后端 reaper SIGHUP→SIGKILL → emit exit。
- **多 tab 终端共存**：每个 tab 独立 toggle；store 用 `Set<sessionID>` 管理；多个 PTY 可以同时活着，互不影响。后端 `terminal_svc.Service.sessions` map 天然支持。

### 键盘快捷键（toggle）

- **`Cmd+\`` (macOS) / `Ctrl+\`` (Windows/Linux)**：toggle 当前 session 的终端面板（等价于点 topbar 的 Terminal 按钮）。复用 VS Code 的约定，目标用户群眼熟。
- 注册位置：chat shell 顶层全局监听 `keydown`（不在 xterm DOM 树内，避免和 terminal-internal key handling 冲突）。
- 拦截规则：
  - 焦点在 chat input / 任意 `<textarea>` / `<input>` 时**仍触发**——是 toggle 不是字符输入，反引号本来就需要 `Shift+\`` 才会被输入到 chat。
  - 焦点在 xterm 容器里时**仍触发**——便于"快进/退出"终端而不必先 blur。
  - 全局快捷键存在期间，session-switch / 没有当前 session 的状态下按下应静默（store 拿不到 sessionID 直接 return）。
- 按钮上要透出 hint：idle 状态 hover tooltip 显示 `终端  ⌘\``；open 状态 toolbar 在关闭按钮前展示 `⌘\`` kbd chip 强化"再按一次就关"的回路。

### 键盘 / 复制粘贴（terminal 内）

- `Cmd+C` / `Ctrl+C`：有选区 → 复制 + `preventDefault`；无选区 → 让 xterm 发 SIGINT。VS Code/iTerm 通用约定。
- `Cmd+V` / `Ctrl+V`：xterm 内置粘贴。
- 滚轮：xterm 默认 scrollback buffer 500 行。

### 视觉

- 主题：light/dark 跟随系统；xterm theme 对象用 `$background` / `$foreground` 现有 token，runtime 读 CSS var 后传给 xterm。
- 字体：`'JetBrains Mono', 'Menlo', 'Monaco', monospace`，13px。
- padding：终端内容区 8px 内边距。

### 错误显示

- `Open()` 失败：toast，回 IDLE。
- 远端 ws 在 OPEN 状态断开：终端顶部红色 banner "连接已断开"，按钮 disabled 直到 ws 重连，PTY 标记 dead。
- PTY 非零退出：toast `shell 退出 (code N)`，自动回 IDLE。

## Error Handling / Edge Cases

### 进程生命周期

| 场景 | 处理 |
|------|------|
| Spawn 失败（shell / cwd / 权限） | `Open()` 透传 error；toast → IDLE |
| PTY 正常退出 | reaper `cmd.Wait()` → emit exit `{code, reason:"natural"}` → map 清 |
| PTY 被外部 kill | 同上，reason=`"killed"` |
| Go 主动 Close | SIGHUP → 200ms grace → SIGKILL；emit reason=`"killed"` |
| app 退出时活 PTY | `service.Stop()` 遍历 map：全部 SIGHUP → 等 500ms → SIGKILL，不阻塞 app 退出 |
| agentred 退出时活 PTY | daemon 同样 reaper；emit reason=`"daemon_shutdown"` |
| 子进程 zombie（user 跑 `nohup foo &` 后 exit） | shell 退出时孤儿被 init 接管，agentre 不管；reaper 只 wait shell |

### 远端 / 网络

| 场景 | 处理 |
|------|------|
| Open 时 device offline | service 查 deviceRegistry，返 `ErrDeviceOffline` |
| Open 时 ws 在线但 daemon 没响应 | 5s timeout → `ErrDaemonTimeout` |
| OPEN 状态 ws 断开 | `remote.Handle` 监听 ws close → emit exit `{reason:"connection_lost"}` |
| ws 重连 | 不自动恢复（远端进程已被 daemon reaper 收）；用户需重新 Open |
| daemon 上 PTY 死了但 ws 还活 | daemon 主动 push `terminal.exit` |
| 高吞吐输出 | 见背压策略：丢旧 + 插 `[--- output throttled ---]` 标记 |

### 并发 / 状态机

| 场景 | 处理 |
|------|------|
| 同 sessionID 连续 Open 两次 | service 用 `sync.Mutex`；第二次 Open 进入临界区前先 Close 第一次 Handle |
| Open in-flight 时被 Close | service 内 `context.WithCancel`；Close 触发 cancel；spawn 已成功则紧接 SIGHUP |
| Write / Resize 在 PTY 已死之后 | 返 `ErrTerminalClosed`，前端 swallow（已在 transition） |
| 前端 EventsOn 漏 unsubscribe | TerminalPanel `useEffect` cleanup 强制 EventsOff；Emit 是 fire-and-forget 不阻塞 |

### 资源 / 安全

| 场景 | 处理 |
|------|------|
| PTY 数量上限 | 不限。MVP 同时活的 PTY ≤ 同时打开的 session 详情数 + agentred 数。app 内 < 50 不会有问题 |
| cwd 不存在 | `exec.Command` 报错；toast 提示 |
| 远端无 shell | spawn 错误透传 |
| `sudo` 输入密码 | PTY 本来就为 isatty 设计，密码输入正常 |

### 数据持久化

**MVP 完全不持久化任何终端数据**：
- 不存 scrollback
- 不写 terminal events 到 SQLite（和 chat message log 是两码事）
- shell 自己的 `.zsh_history` / `.bash_history` 当然会写（PTY 默认行为，不是 agentre 控制的）

这是和 chat session 反着来的设计——chat 的 turn 全部 audit，terminal 完全 ephemeral。理由：终端是用户的私人临时空间，audit 反而引入隐私问题（密码、token 误粘贴会留痕）。

## Open Questions / 后续可扩展

- **后台保活**：如果用户实际使用中频繁踩"切走 session 就死"的坑，再加一个 "detach" 状态（PTY 在但 UI 不挂），需要额外的 PTY map TTL / 用户显式 "keep alive" 操作。
- **多终端**：xterm tab bar 在终端内，开多个 PTY。MVP 不做，但 Wails service 的 sessionID key 可以扩成 `(sessionID, terminalIdx)`。
- **Windows / ConPTY**：creack/pty 在 Windows 上是 ConPTY wrapper 但有 edge case（颜色、resize 时机）。延后到有需求再补 build tag + 分别测。
- **Local echo**：远端 latency 在 LAN 场景可忽略；如果未来跑 cloud relay 时延 > 100ms，再考虑加（要判断 raw mode，复杂）。
- **scrollback 持久化**：仅当用户明确反馈需要（例如想搜索过去 10 分钟的输出）再考虑，且要先想清隐私问题。
