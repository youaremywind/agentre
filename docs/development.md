# Development Conventions

Hard constraints for testing, fixing bugs, and designing interfaces. **Red → Green → Refactor, no exceptions.**

## TDD / BDD Workflow

1. **Red — write the failing test first.**
   - **New feature:** start from a **BDD-style** behavior spec — describe the user-visible result (`Given … When … Then …` or a goconvey `Convey("when X, then Y")` block). Happy path + at least one boundary/error case.
   - **Bug fix:** write a **TDD** regression test that reproduces the failure (same error/assertion). Run it and confirm it fails for the right reason. If the bug genuinely cannot be reproduced in a test, **say so explicitly** before patching.
   - Tests sit next to the code (`foo_test.go`); use table-driven cases for branch combinations; mocks go through `go.uber.org/mock`, placed in `mock_*/` subdirectories (generated by `make mock`).
2. **Green — the minimal code to make the test pass.** No speculative generalization, no reserved fields. Run `make test` and confirm red turns green without regressing anything else.
3. **Refactor — clean up under the green light.** Rename / extract functions / deduplicate. Run `make test` and `make lint` again before committing.

**Layer-by-layer testing:** repository tests cover persistence + boundaries (use `testutils.Database(t)` sqlmock, **assert SQL/parameters**, don't spin up a real DB); service tests cover business rules with the repo mocked via `mockgen`; `internal/app/` bindings are too thin to unit test — **don't stuff testable logic into `App`.**

**Coverage:** target ≥80% for the service/repository layers of new packages. Check with `make test-cover`.

## Commits

Start the commit message with a **gitmoji emoji character** (write the emoji glyph directly, **do not** substitute a shortcode like `:sparkles:` or an English word), then a space followed directly by the change description, no module / scope prefix needed:

```text
✨ add chat tab drag reorder
🐛 fix session cwd restore
✅ test provider missing API key
📝 update commit message discipline
```

Issue / PR suffixes are selective, not automatic. Add an issue suffix only when the commit is a single focused change that directly resolves or advances a known issue, using the issue number form such as `#123`. When the work is organized around a PR, do **not** append the PR number to every commit; the PR already carries that context. In general, prefer issue references over PR references, and leave the suffix off when there is no relevant issue.

Common gitmojis and the category they map to when the changelog is generated (release flow):

| gitmoji | Purpose | changelog category |
|---------|------|---------------|
| ✨ | New feature | 🚀 Major new features |
| 🐛 | Bug fix | 🐛 Bug fixes |
| ⚡️ | Performance optimization | ⚡️ Performance optimizations |
| ♻️ | Refactor / compatibility | ♻️ Refactoring and compatibility |
| 🎨 | UI improvement | 🎨 UI improvements |
| 📝 | Docs / changelog | Other |
| ✅ | Tests | Other |
| 🔧 | Configuration | Other |
| 🔒 | Security | Other |
| 🔖 | Release / version bump | — (the release commit itself) |

Release commits use fixed forms: `🔖 release v{version}` (version bump + `CHANGELOG.md`) and `📝 update changelog for v{version}` (sync the documentation-site changelog).

## SOLID

Run through this for every new package / type / function before merging:

- **S — Single Responsibility.** One reason to change. A service doing parsing + persistence + notification at once → split it. A 300-line function almost always violates SRP.
- **O — Open/Closed.** Extend by adding a new type/strategy, don't edit a switch scattered across 10 callers. Adding an Agent backend (Claude Code / Codex / built-in) should be implementing an interface, not patching `if engine == ...`.
- **L — Liskov.** An implementation must honor the interface contract (nil semantics, error types, side-effect boundaries). No "this implementation ignores ctx" surprises.
- **I — Interface Segregation.** Define narrow interfaces at the **consumer** (`service` declares what it needs from the repo), don't build a big fat interface at the producer. A consumer that needs 2 methods shouldn't depend on a 20-method interface.
- **D — Dependency Inversion.** `service/` depends on the repository **interface**, never on the concrete implementation. The concrete implementation is wired up in `main.go` / `internal/app/` — this is the prerequisite for TDD being able to mock.

## High Cohesion, Low Coupling

How SOLID takes concrete shape in this repository — check against this before opening a PR:

**High cohesion (a single reason to be grouped together)**

- One package set per domain: `model/entity/<domain>_entity` + `repository/<domain>_repo` + `service/<domain>_svc`. Open a new package for a new domain, **don't** pad an adjacent old package with the feature.
- Rich domain entities carry single-entity logic (`Check(ctx)` / `IsActive()` / `GetXxx/SetXxx`); the service only does cross-entity coordination and orchestration of external dependencies. Push rules down into the entity so the service doesn't bloat.
- One Wails binding file per domain (`internal/app/<domain>.go`), where the method just does parse → `svc.Xxx().Method` → return.
- Cross-cutting capabilities (`diff` / `jsonrpc` / `keychain` / `paths` / `remotefs` …) live in `internal/pkg/<concern>`, each package with a single concern and self-contained; don't scatter them into some domain.

**Low coupling (only recognize interfaces across boundaries)**

- Dependencies are one-way: `internal/app → service → repository → model/entity`. `internal/pkg` is a leaf cross-cutting layer, **referenced by every layer but never reverse-imports** `service` / `repository` (there are currently zero reverse dependencies, hold that line).
- DIP: `service` depends only on the `repository` **interface**; the concrete implementation is wired up via `RegisterXxx(impl)` in bootstrap/main; this is exactly what lets service unit tests inject mockgen mocks.
- ISP: interfaces are defined at the **consumer** and narrowed as needed; a consumer that needs 2 methods shouldn't depend on a 20-method fat interface.
- Cross-package collaboration goes through accessors (`xxx_repo.Xxx()` for the repository, `xxx_svc.Default()` for another service); don't `new` someone else's implementation, and don't reach into another domain's tables directly via `db.Ctx`.
- Don't skip layers: `internal/app` doesn't touch `repository` / `db` directly (the binding layer only recognizes the service), and a service doesn't bypass its own repo to assemble raw SQL.
- Boundary encapsulation: the front end and back end only go through the Wails binding (no HTTP-style app API is added); remote-execution details are locked behind the `internal/daemon/client` interface, and the upper-layer `chat_svc` is unaware of subprocess / connection details.

## Fix Discipline (hard constraint)

When a bug / regression / abnormal behavior appears:

1. **Write the test first, confirm the problem exists.** Write a failing test and run it. If it doesn't fail (or fails for the wrong reason), you don't yet understand the bug — investigate before patching. No fix lands without a red-then-green test pair.
2. **A clean fix — fix the root cause, don't patch over it.** Fix the producer of the bad value, not every consumer. **Don't** add `if x == nil` to mask a caller bug. **Don't** repeat the same field's normalization at three call sites — normalize once at the boundary. A `// workaround because X returns Y` comment is a smell; the code beneath it probably needs changing.
3. **Don't touch anything unrelated.** The diff only touches the producer, its test, and at most one obviously-in-scope drift right under your cursor. **No** drive-by refactors / rename sweeps / formatter passes / dead-code cleanups / import reordering / unrelated test churn — they bury the real fix and break `git bisect`. Multi-day refactors / hot subsystems / design problems → flag them and ask first.
4. **Reuse first.** Before adding a new component / hook / util / Go helper, grep for an existing one. Copying >10 lines? Extract it. Two near-identical blocks getting the same bug fix? The second one is the bug — delete it and call the first.

## Test Stack

- **Framework** — `github.com/smartystreets/goconvey/convey` (BDD nesting) + `github.com/stretchr/testify` (assertions) + `go.uber.org/mock` (interface mocks) + `github.com/DATA-DOG/go-sqlmock` (DB mock). Use table-driven cases for branch combinations.
- **Repo unit tests: always go through `testutils.Database(t)`, never use a real SQLite.** Take the `(ctx, _, mock)` triple; the business code automatically hits the mock via `db.Ctx(ctx)`. Each case asserts SQL and parameters precisely with `mock.ExpectQuery / ExpectExec`, and always ends with `assert.NoError(t, mock.ExpectationsWereMet())`. Note that **`testutils.Database` uses the MySQL dialect**, so use `` `table_name` `` backticks in the regex match; GORM `Create / Save / Update` opens a transaction automatically by default, so the match template writes `ExpectBegin / ExpectExec / ExpectCommit` (add `ExpectRollback` for the error path). **Exception**: migrations themselves (`migrations/*`) + end-to-end startup tests like `internal/bootstrap/cago_test.go` may use `t.TempDir()` to spin up a real SQLite and verify the full DDL/data flow. Every other repo / service test goes through a mock.
  - **Historical lesson**: the early `agent_backend_repo_test.go` used `t.TempDir()` + real SQLite, testing SQLite dialect side effects rather than the SQL the repo itself assembled; new repo tests must use sqlmock from the start.
- **Service unit tests** — `mockgen` generates the `repository` mock, injected via `RegisterXxx(mockRepo)`. The service depends only on the repository interface (DIP), so it needs no DB; **do not spin up sqlmock or a real DB in service tests**.
- **Mock generation** — add `//go:generate mockgen -source xxx.go -destination mock_xxx_repo/mock_xxx.go` at the top of the repository interface, and run `make mock` uniformly.

### Test Organization

- One `setupXxxTest(t)` helper inside each `*_svc` / `*_repo` package: it constructs `gomock.NewController(t)` + `t.Cleanup(ctrl.Finish)`, injects the mock, and returns `(ctx, mocks..., subject)`.
- Extract shared operations into `t.Helper()` functions, so an assertion failure points at the caller rather than inside the helper.
- **GoConvey nesting convention:** top level = feature name / method name (`Convey(..., t, func() { ... })`) → nested = scenario (success / failure / boundary) → deeper = temporal behavior ("after doing A, then do B"). Each `Convey` block runs independently, so you can safely set up the mock in the outer layer.
- Component state (`testutils.Cache()` / `testutils.Redis()` / `iam.SetDefault(...)`) has a `sync.Once` cache and **leaks across cases**. For IAM / cache setup, manually `SetDefault` to re-inject inside the helper, to avoid the mock handle being held by a stale reference.

### Linter Exceptions (inline `//nolint`; `.golangci.yml` only configures the enabled list and global excludes)

These exceptions are inline `//nolint` comments written in the `.go` files, not in `.golangci.yml` (which only enables the corresponding linters + globally excludes `frontend/` + has gosec G-code excludes). cago tool handlers wrap the error into a `tool.ErrorResult` and then return `(*ToolResultBlock, nil)`, so the LLM sees a failed call but doesn't abort the whole turn — preserve this return shape, and add a `//nolint:nilerr` comment at the return point explaining why (e.g. `internal/service/llm_provider_svc/llm_provider.go`, `project_svc/project.go`, `remote_device_svc/refresh.go`). The fake `*http.Response` body in `llm_provider_svc/*_test.go` and `internal/daemon/*_test.go`, and the cert path derived from `t.TempDir()`, use inline `//nolint:bodyclose` / `//nolint:gosec` to waive the checks.

## Logging (mandatory on critical paths)

cago's zap wrapper `github.com/cago-frame/cago/pkg/logger` writes to `<AppDataDir>/logs/agentre.log` (drops to debug+ once you turn on "Settings → Version & Update → Debug logging"). For the command list to read logs, query SQLite, and reproduce production bugs, see [debugging.md](debugging.md).

- **How to call:** use `logger.Ctx(ctx)` when you have a ctx, otherwise `logger.Default()` (in bootstrap / `gogo.Go`). Fields are always structured via `zap.Xxx(...)`; **do not** use `fmt.Sprintf` to assemble the message.
- **Message format:** lowercase + `package.Method:` prefix, dynamic values in fields, to make it greppable. Example:

  ```go
  logger.Ctx(ctx).Warn("chat_svc.Stop: runner.Abort failed",
      zap.String("session_id", sid), zap.Error(err))
  ```

- **Mandatory log points:**
  1. Lifecycle boundaries (session/turn/runtime start-stop, abort, auto-continue)
  2. External call boundaries (CLI spawn/exit, remote borrow/return, HTTP, migrations, FS)
  3. Degradation / fallback / retry
  4. Before a service/repo `return err`
  5. State changes (permission mode, login, remote connection)
- **Levels:** Debug = normal-path detail; Info = business milestone; Warn = recoverable error / swallowed defer error; Error = state may be corrupted, fields must be sufficient to reconstruct the scene.
- **Anti-patterns:** only `return err` without logging; spamming info inside a loop; cramming dynamic values into the message.

## Windows subprocesses

Agentre is a Windows GUI app, so background `exec.Command` subprocesses can flash a black console window unless their process attributes explicitly suppress it. Any Windows background subprocess that is not intentionally user-visible must call `procattr.ApplyNoConsoleWindow(cmd)` before `Start` / `Run` / `Output` / `CombinedOutput`.

This is mandatory for agent backend CLI processes (`claude`, `codex`, `pi`) because they may launch once per turn. Do not duplicate raw `syscall.SysProcAttr{HideWindow: true, CreationFlags: CREATE_NO_WINDOW}` at each call site; use `internal/pkg/procattr` so the rule is test-covered and stays consistent.
